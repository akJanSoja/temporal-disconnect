Temporal Disconnect

The structural lag between the onset of a life event and its complete expression in recorded transaction data.

Most transaction models ask: when did money move?
The more useful question is: what was happening in this person's life?
The gap between these two questions is Temporal Disconnect.
The data is not wrong. The framing is wrong. Transactions are timestamped. The events that drive them are not. This is not a data quality problem. The data is accurate. It is a framing problem.

SQL Pattern: Assign Event Windows

Using window functions to scan forward from anchor transactions and assign EventWindowKey. This runs in the staging/transformation layer.

SQL — Window function pattern 
-- Step 1: Identify anchor transactions
WITH anchors AS (
  SELECT
    customer_id,
    transaction_date  AS anchor_date,
    category,
    ROW_NUMBER() OVER (
      PARTITION BY customer_id
      ORDER BY transaction_date
    ) AS event_seq
  FROM transactions
  WHERE category IN ('Travel_Booking', 'Sporting_Goods', 'Real_Estate', 'Auto_Dealer')
),

-- Step 2: Build event windows around each anchor
windows AS (
  SELECT
    a.customer_id,
    a.anchor_date,
    a.event_seq,
    a.anchor_date - INTERVAL '30 days'  AS window_start,
    a.anchor_date + INTERVAL '180 days' AS window_end
  FROM anchors a
),

-- Step 3: Classify each transaction by event stage
staged AS (
  SELECT
    t.*,
    w.event_seq,
    w.anchor_date,
    CASE
      WHEN t.transaction_date < w.anchor_date
        THEN 0  -- Pre-signal
      WHEN t.transaction_date <= w.anchor_date + INTERVAL '30 days'
        THEN 1  -- Initiation
      WHEN t.transaction_date <= w.anchor_date + INTERVAL '90 days'
        THEN 2  -- Active
      WHEN t.transaction_date <= w.anchor_date + INTERVAL '150 days'
        THEN 3  -- Normalization
      ELSE 4     -- Post-event
    END AS event_stage
  FROM transactions t
  JOIN windows w
    ON  t.customer_id = w.customer_id
    AND t.transaction_date BETWEEN w.window_start AND w.window_end
)

SELECT * FROM staged
ORDER BY customer_id, transaction_date;

Does This Scale?

Event window logic is a range join. It scales the same way any temporal join scales — with partitioning, staging-layer assignment, and the right engine.

Partitioning

Partition on CustomerKey before windowing to avoid full table scans. The anchor table is small; the transaction table is large. Classic broadcast-join pattern.

Delta Lake / Fabric

Assign EventStage in the staging layer as a calculated column. Never in DAX — row-context temporal lookups do not scale in a semantic model.

Spark

Broadcast the anchor table if < 1 GB. Handle skewed customers (high transaction volume) with salted joins.

Use Cases in Practice

Temporal Disconnect is domain-agnostic. Any dataset where behavior precedes or extends beyond the timestamps that record it contains this structure. Below are four domains where applying the framework produces measurable analytical lift.

Fraud detection

Detect the Ramp-Up, Not the Loss

Fraud rarely happens in isolation. A fraudulent account typically shows a behavioral ramp-up before the fraud event — small test transactions, category shifts, velocity changes. A naive model flags the fraud after the fact. A TD-aware model detects the pre-signal stage and scores the account before the loss occurs.

What changes

  • Detect account takeover from Stage 0 pre-signal: login pattern changes, small test charges, new merchant categories
  • Separate legitimate life-event spend spikes (moving, travel) from anomalous spikes — reducing false positive rates dramatically
  • Score transactions not just on amount and category but on where in an event window they fall — a large transaction in Stage 2 of a detected travel event is expected; the same transaction with no event context is suspicious
  • Early intervention: flag accounts in Stage 1 of an anomalous pattern rather than waiting for Stage 2 peak
Example: An account makes three small transactions at a new electronics merchant (Stage 0 pre-signal), then a large transfer to a new payee (Stage 1 anchor), then rapid cash withdrawals across three days (Stage 2 active). A naive model sees three unrelated events. TD sees a card-testing → takeover → cashout sequence.
Churn & LTV

Churn Starts 60 Days Before Cancellation

Churn does not happen on the day a customer cancels. It begins 60–90 days earlier as a behavioral shift — reduced engagement, category contraction, spend migration to competitors. The cancellation is Stage 3. The detectable signal is Stage 0.

What changes

  • Identify pre-churn Stage 0 signal: declining transaction frequency, shrinking category breadth, first use of a competitor merchant
  • Trigger retention interventions at Stage 1, not after cancellation
  • Build LTV models on event-adjusted spend — a customer in the middle of a moving event has temporarily suppressed spend, not permanently reduced LTV
  • Distinguish life-event churn (customer moved abroad, genuinely left) from competitive churn (customer is still there but spending elsewhere)
Example: A customer’s monthly grocery spend drops 40% over two months. Naive model: LTV decline, possible churn risk. TD model: anchor detected — first transaction at a new supermarket chain two months ago. Customer moved neighborhoods. Not a churn signal. A retention campaign here wastes budget and annoys the customer.
Supply chain

The Decision Precedes the Order

B2B procurement follows the same event-window structure as consumer spend. A buyer’s decision to switch supplier, ramp production, or enter a new market generates a predictable sequence of purchase orders across weeks or months. The decision precedes the transaction data by the entire lead time.

What changes

  • Detect supplier switching in Stage 0 before the first competitive order is placed — signals: RFQ activity, sample orders, logistics inquiries
  • Forecast seasonal demand surges from Stage 1 initiation signals rather than waiting for Stage 2 peak orders to arrive
  • Identify customers building strategic inventory vs. steady-state ordering — they look identical in a naive model but require different account management
  • Route capacity planning decisions earlier by detecting event windows across the customer base simultaneously
Example: A distributor notices a mid-size customer placing small orders of a new product category in Q1. Naive model: minor upsell opportunity. TD model: Stage 0 pre-signal of a product line expansion. By Q3, that customer’s order volume has tripled. Sales had the data in Q1.
Marketing

Same Transaction, Different Context

Segment instability is expensive — customers churn between segments month to month, making targeting inconsistent and ROI hard to measure. The root cause is that naive segments are based on timestamped spend, not on the life event that drives it. A customer buying baby furniture is not in the “home & garden” segment. They are in Stage 2 of a new baby event.

What changes

  • Segment by event type + event stage rather than spend category
  • Campaign timing: reach customers in Stage 1 (initiation) when they are actively deciding, not Stage 2 (active) when they have already committed
  • Reduce segment churn: a customer stays in the same event-stage segment for the duration of their window — 60–180 days of stable targeting instead of monthly reshuffling
  • Personalization: tailor messaging to stage, not just category. Stage 1 messaging: help them decide. Stage 3 messaging: help them wind down and transition to the next normal
Example: Two customers both spend 8 000 kr at Stadium in the same week. Customer A: no event context detected — standard sporting goods segment. Customer B: anchor detected 6 weeks ago — multi-passenger flight booking, children’s ski gear. Customer B is in Stage 2 of a family sports holiday. Same transaction, completely different context, completely different next best action.

In each domain, the underlying mechanism is the same: a decision is made before the data reflects it, and the data continues to reflect it after the decision has been acted on. Modeling the event rather than the transaction is not a feature addition. It is a different question.

Who Benefits from Temporal Disconnect Analysis

Temporal Disconnect is not a banking concept or a retail concept. It appears in any dataset where a decision precedes its transactions — or where behavior continues after the event that caused it. Below are twenty organisations across four domains where applying the framework produces a measurable analytical advantage.

Fraud & abuse detection
  • Banks & card issuers — Card-testing sequences leave a pre-signal pattern of small, low-risk transactions at new merchants before a large fraudulent purchase is attempted. The fraud event is Stage 2. The detectable window opens at Stage 0.
  • Grocery retailers — Loyalty discount abuse follows a temporal arc. Customers gaming promotional windows split baskets, time purchases to activation dates, and cluster spend around campaign periods in ways that are invisible transaction-by-transaction but obvious as an event sequence.
  • B2B distributors — Salespeople offering discounts in a pattern correlated with quota periods rather than customer negotiation is a TD signal finance teams rarely model. The anchor is the quota deadline. The pre-signal is the discount pattern in the 30 days before it.
  • Insurance companies — Fraudulent claims are rarely isolated. Policyholder behavior in the 60–90 days before a claim — new coverages added, valuation inquiries, address changes — forms a detectable pre-signal stage that a naive claims model never sees.
  • E-commerce platforms — Return abuse follows a buy → use → return sequence clustered around identifiable events: concerts, weddings, job interviews. Each return looks legitimate in isolation. As an event window, the pattern is structural and repeatable.
Churn & retention
  • Grocery chains (ICA, Coop, Axfood) — Basket migration to a competitor begins 8–12 weeks before a loyalty card goes dormant. Category breadth contracts first — fewer departments per visit — then visit frequency drops. By the time spend falls, the customer has already decided.
  • Telecom operators — A data usage drop combined with an increase in support contacts is a pre-churn Stage 0 signal that appears reliably 60 days before a port-out request. Intervening at Stage 1 costs a fraction of winning the customer back after they have left.
  • SaaS companies — Feature usage contraction precedes cancellation by a predictable arc. Users stop using advanced or collaborative features first, then revert to core functions only, then go dark. Each step is a stage. The cancellation is Stage 3. The recoverable window is Stage 0–1.
  • Fuel card providers (B2B) — A fleet customer reducing fill frequency and shifting to spot purchases rather than contracted volume is in Stage 0 of a supplier switch. The contract renewal conversation is Stage 3. TD moves that conversation to Stage 1.
  • Gyms & fitness brands — Attendance decline follows a consistent six-week pre-churn arc before a membership lapses. Class booking drops before visit frequency drops. Visit frequency drops before the membership is cancelled. Each step is measurable and actionable.
Supply chain & B2B procurement
  • Industrial distributors (Würth, Ahlsell) — A customer placing small trial orders of a new product category is in Stage 0 of a strategic volume decision. The anchor is the first category-crossing order. By the time volume arrives, a TD-aware account manager has already had the conversation.
  • Food manufacturers — Retailer order patterns ahead of seasonal campaigns are predictable event windows. Orders ramp 10–14 weeks before a campaign launch, peak during the campaign, and normalize over 4–6 weeks afterward. Capacity planning that ignores this window is reacting to Stage 2 data.
  • Pharma wholesalers — Hospital purchasing shifts in category mix and order size before a contract renewal signal switching intent. A TD model on procurement data surfaces this 90–120 days before the tender — when retention is still possible.
  • Construction material suppliers — Project-driven procurement follows a multi-stage event window that can span 6–18 months. The anchor is often a permit application or architectural order. The cascade — structural materials, finishing materials, fixtures — is entirely predictable if the anchor is detected early.
  • Auto parts distributors — Workshop order patterns shift before a fleet tender. Test orders in new categories precede volume by 4–8 weeks. A naive model sees low-value orders. A TD-aware model sees a customer in Stage 1 of a supplier evaluation.
Marketing & personalisation
  • Department stores (Åhléns, NK) — Life events activate distinct category sequences. A new baby, a move, a wedding each produce a recognisable spend arc across 3–6 months. Customers in Stage 1 are the highest-value targets for personalised campaigns — and the easiest to over-communicate to if the stage is not modelled correctly.
  • DIY retailers (Byggmax, Hornbach) — Renovation events have a clear anchor — a large lumber or tool purchase — and a long cascade of finishing spend: paint, fixtures, lighting, décor. The cascade continues for months after the anchor and is entirely predictable in category sequence.
  • Travel agencies & OTAs — The booking date is the anchor. Everything that follows — luggage, travel insurance, currency exchange, airport parking, destination activities — is Stage 1–2 spend that is predictable to the day relative to the departure date. Campaigns timed to the event stage outperform campaigns timed to the calendar.
  • Car dealers & leasing companies — The pre-purchase research phase leaves spend signals 3–6 months before a vehicle purchase: driving school top-ups, parking apps, fuel stops in new locations, insurance comparison sites. The anchor — the purchase — comes last. TD moves the commercial conversation earlier.
  • Streaming & media (Spotify, Netflix) — Content consumption patterns shift before a subscription change. Users about to upgrade consume more content in more categories. Users about to downgrade narrow their consumption to one or two genres. Both are Stage 0 signals with measurable commercial responses.

In every case above, the underlying data is accurate. The transactions are timestamped correctly. The problem is not data quality — it is that timestamps record when money moved, not what was happening. Temporal Disconnect gives every transaction a second dimension: not just when, but where in the arc of a human or commercial decision.
